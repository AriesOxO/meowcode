# 从零开始做 AI 编码助手（一）：技术选型与架构设计

## 为什么要做这个项目

用了一段时间 Claude Code，深深被 AI 编码助手的能力震撼。但也有一些想法：
- 能否支持更多模型？（DeepSeek 性价比太高了）
- 能否更轻量？（专注核心功能）
- 能否自己控制？（开源、可定制）

于是决定自己做一个 **MeowCode**。

## 目标

不是要替代 Claude Code（那是不可能的），而是：
1. **深入理解 AI Agent 的设计原理**
2. **做一个够用的编码助手**（满足自己日常需求）
3. **支持多模型**（DeepSeek、智谱、Claude 等）
4. **记录整个过程**（边做边学边写）

## 技术选型

### 编程语言：Python

考虑过 Go、Rust，但最终选择 Python：

**优点**：
- ✅ AI 生态最成熟（OpenAI SDK、Anthropic SDK 都是 Python 优先）
- ✅ 开发速度快（1-2 周就能做出 MVP）
- ✅ 代码简洁（标准库足够强大）
- ✅ 打包方案成熟（PyInstaller / Nuitka）

**缺点**：
- ⚠️ 打包体积大（20-40MB）
- ⚠️ 启动速度慢（1-2秒）

但对于 MVP 来说，开发效率 > 运行效率。

### LLM 提供商：DeepSeek 起步

**为什么选 DeepSeek？**

价格对比（修复 100 个 Bug 的成本）：
- DeepSeek v4-flash: ¥0.18
- GLM-4-flash: ¥0.15
- Claude Sonnet 4.5: ¥54（贵 300 倍！）

作为个人开发者，成本是关键因素。DeepSeek 性价比无敌。

**未来会支持**：
- 智谱 GLM（备选方案）
- Anthropic Claude（高端场景）
- OpenAI GPT（兼容性考虑）

### 界面：CLI

选择命令行界面（CLI）而不是 GUI：
- ✅ 简单高效
- ✅ 开发成本低
- ✅ 适合开发者使用场景
- ✅ 易于自动化和集成

## 架构设计

### 核心循环

```
用户输入任务
  ↓
┌─────────────────┐
│  查询 LLM       │
└────────┬────────┘
         ↓
┌─────────────────┐
│  解析 Action    │ → bash / read_file / write_file / exit
└────────┬────────┘
         ↓
┌─────────────────┐
│  执行 Action    │
└────────┬────────┘
         ↓
┌─────────────────┐
│  反馈结果       │
└────────┬────────┘
         │
         └─→ 循环，直到完成或超时
```

### 多模型架构

使用**抽象工厂模式**：

```python
# 抽象基类
class BaseLLMProvider:
    def query(messages) -> str
    def get_model_info() -> dict

# 具体实现
class DeepSeekProvider(BaseLLMProvider):
    ...

class ZhipuProvider(BaseLLMProvider):
    ...

# 工厂函数
provider = create_provider("deepseek", api_key)
```

**好处**：
- ✅ 新增模型只需实现 3 个方法
- ✅ 切换模型只需修改配置
- ✅ 统一接口，代码清晰

### 目录结构

```
meowcode/
├── agent.py              # 主入口
├── models/               # LLM 提供商
│   ├── base.py          # 抽象基类
│   ├── deepseek.py
│   └── factory.py
├── tools/                # 工具系统
│   ├── execute.py       # 命令执行
│   └── file_ops.py      # 文件操作
├── utils/                # 工具函数
│   ├── parser.py        # Action 解析
│   └── config.py        # 配置加载
└── config/               # 配置文件
    └── config.yaml
```

模块化设计，职责清晰。

## 感想

做技术选型时最大的收获是：**不要过度设计**。

一开始想用 Go 做（性能好、打包小），但想想 MVP 阶段最重要的是**快速验证想法**。Python 的开发效率让我 2 天就完成了核心框架，如果用 Go 可能要 1 周。

**工具是为目标服务的，不是目标本身。**
